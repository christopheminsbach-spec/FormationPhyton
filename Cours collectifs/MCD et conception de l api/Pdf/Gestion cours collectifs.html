Fonctionnalités couvertes par le MCD

Le modèle conceptuel de données COURS permet de répondre aux besoins principaux de l’API de gestion des cours collectifs du studio.

1. Création d’un cours

L’entité COURS permet d’enregistrer un nouveau cours avec les informations suivantes :

  • Nom du cours

  • Nom du coach

  • Capacité maximale

  • Prix du cours

Lors de la création, le nombre d’inscrits est automatiquement initialisé à 0.

2. Affichage des cours

Grâce aux données enregistrées dans l’entité COURS, l’API peut récupérer la liste des cours disponibles et afficher :

  • Identifiant du cours

  • Nom du cours

  • Coach

  • Nombre de places occupées

  • Nombre de places restantes

  • Prix

  • Chiffre d’affaires actuel

  • Chiffre d’affaires potentiel

3. Ajout d’une inscription

Le champ Nombre_Inscrits permet de suivre le nombre de participants.

Lorsqu’une personne s’inscrit :

  • Vérification de l’existence du cours

  • Vérification des places disponibles

  • Incrémentation du nombre d’inscrits

Exemple :

Avant inscription :
Nombre_Inscrits = 5
Capacite_Max = 10

Après inscription :
Nombre_Inscrits = 6
Capacite_Max = 10

4. Affichage des cours complets

L’API peut identifier les cours complets grâce à la règle :

Nombre_Inscrits = Capacite_Max

Exemple :

Pilates
Places : 10 / 10
Statut : Complet

5. Suppression d’un cours

L’identifiant Id_Cours permet de retrouver un cours précis afin de le supprimer de la base de données.

Exemple :

DELETE /courses/3

Supprime le cours dont l’identifiant est 3.

6. Gestion avec SQLAlchemy et SQLite

Le MCD est transformé en modèle de données exploitable par l’ORM SQLAlchemy.

Correspondance :

MCDSQLAlchemySQLite
Entité COURSClasse Python CourseTable courses
Id_CoursAttribut idClé primaire
Nom_CoursAttribut nameColonne VARCHAR
Nom_CoachAttribut coachColonne VARCHAR
Capacite_MaxAttribut capacityColonne INTEGER
Nombre_InscritsAttribut registered_countColonne INTEGER
PrixAttribut priceColonne FLOAT

Conclusion :

Le MCD proposé est volontairement simple et adapté à une première version d’API. Il permet de gérer entièrement le cycle de vie d’un cours : création, consultation, inscription, contrôle des places et suppression, tout en étant compatible avec une architecture FastAPI + SQLAlchemy + SQLite.

SQL

    cours.id_cours,
    cours.nom_cours,
    cours.capacite_max,
    COUNT(inscription.id_inscription) AS inscrits
FROM cours
JOIN inscription 
ON cours.id_cours = inscription.id_cours
GROUP BY cours.id_cours
HAVING COUNT(inscription.id_inscription) >= cours.capacite_max;

Cas d'utilisation : Garder l’historique des inscriptions

Acteur

Administrateur du studio / Utilisateur

Objectif

Conserver une trace de toutes les inscriptions effectuées afin de connaître l’historique des participants, des cours suivis et des dates d’inscription.

Principe

Au lieu de supprimer définitivement une inscription lorsqu’un cours est supprimé ou lorsqu’une inscription est annulée, l’application conserve les informations dans la table INSCRIPTION.

L’historique permet de répondre aux questions :

  • Qui a participé à quel cours ?

  • À quelle date la personne s’est-elle inscrite ?

  • Quel était le statut de l’inscription ?

  • Combien d’inscriptions ont été réalisées sur un cours ?

Règles de gestion

RG1 : Une inscription est conservée après sa création.

Exemple :

Marie Dupont
→ Yoga débutant
→ Inscription le 15/06/2026
→ Statut : Confirmée

Cette information reste enregistrée.

RG2 : Une inscription possède un statut.

Valeurs possibles :

StatutSignification
ConfirméeParticipant inscrit au cours
AnnuléeInscription annulée mais conservée
TerminéeCours réalisé

RG3 : La suppression d’un cours ne supprime pas forcément l’historique.

Deux solutions possibles :

Solution recommandée : suppression logique

On ajoute un champ :

Actif

dans la table COURS.

Exemple :

Avant :

Cours : Yoga débutant
Actif : Oui

Après suppression :

Cours : Yoga débutant
Actif : Non

L’historique reste visible.
